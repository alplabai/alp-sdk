# websocket-frame

libwebsockets teaching example. Masks and encodes a single
unfragmented RFC 6455 TEXT frame into a buffer, then decodes
(unmasks) it back out and asserts the payload survives the round
trip. No socket, no TLS, no handshake -- this is the frame-layout
byte accounting you'd want to get right before ever opening a real
connection.

**[UNTESTED]** -- both in the "native_sim only, no bench" sense every
example in this batch carries, AND more specifically: the real
libwebsockets library is not linked here. See "Integration reality"
below.

## Integration reality

`board.yaml` declares `libraries: [libwebsockets]`, which is enough
for `scripts/alp_project.py` to emit `CONFIG_ALP_LWS_NO_TLS=y` (the
SW-fallback default from
`metadata/library-profiles/libwebsockets/hw-backends.yaml`). It does
**not** get the real library linked in, for two compounding reasons
(both confirmed by reading the actual checkout at
`modules/lib/libwebsockets` -- unlike `coremqtt_sn`, this module
fetch *did* succeed and the full ~230 KB upstream source tree is
present):

1. libwebsockets ships no `zephyr/module.yml`, so west's module
   scanner never adds its include path or CMakeLists.txt to a Zephyr
   build automatically -- an app has to wire that in by hand.
2. Even with the include path added, `include/libwebsockets.h`
   unconditionally includes `lws_config.h` (its line 42), which does
   not exist in the raw checkout -- it's generated by libwebsockets'
   own `CMakeLists.txt` `configure_file()` step, gated on OpenSSL/
   mbedTLS/zlib discovery and dozens of `LWS_HAVE_*` feature probes.
   Running that full configure is more than "a small local CMake
   source-compile" for a teaching example (see the work plan) --
   out of scope here.

Even past the headers: libwebsockets' only public framing entry point,
`lws_write()` (`include/libwebsockets/lws-write.h`), takes a live
`struct lws *wsi` and may only be called from that connection's
`WRITEABLE` callback. There is no standalone function that masks and
frames a buffer offline, independent of a running `lws_context` and
an established connection -- so even a from-scratch, no-TLS build
would still need a live event loop and socket, not just a
source-compile.

`src/main.c` therefore hand-rolls the exact RFC 6455 section 5.2
frame layout libwebsockets produces internally
(`ws_frame_encode_text()` / `ws_frame_decode_text()`), instead of
calling into the library.

To swap to the real library once a full CMake-configured
libwebsockets build is wired into the app: allocate `unsigned char
buf[LWS_PRE + payload_len]`, fill `&buf[LWS_PRE]` with the payload,
and call `lws_write(wsi, &buf[LWS_PRE], payload_len,
LWS_WRITE_TEXT)` from `LWS_CALLBACK_CLIENT_WRITEABLE` -- lws computes
the mask key and frame header into the reserved `LWS_PRE` bytes ahead
of your payload, then sends it out the wsi's socket. `LWS_PRE` (16
bytes on most targets) and `LWS_WRITE_TEXT` (`enum
lws_write_protocol`'s value `0`) are both defined in
`include/libwebsockets/lws-write.h`, matching this example's opcode.

## What this shows

* The RFC 6455 5.2 frame layout for a short (<126-byte), masked,
  unfragmented TEXT frame: `FIN|RSV|opcode` byte (`0x81`),
  `MASK|len(7)` byte, a 4-byte masking key, then the XOR-masked
  payload.
* Why client-to-server WebSocket frames are masked at all (RFC 6455
  5.1/5.3) -- a security requirement, not a checksum -- and why a real
  implementation (libwebsockets included) picks that key from a CSPRNG
  per frame rather than the fixed demo key used here.
* A buffer-only encode/decode round trip with an explicit
  byte-for-byte payload assertion.

## Build

```bash
# Standalone, native_sim (host binary; no hardware needed):
west build -b native_sim/native/64 examples/connectivity/websocket-frame \
    -- -DEXTRA_ZEPHYR_MODULES=$(pwd)
west build -t run
```

## Expected output

```
[websocket-frame] encoding TEXT frame: payload="hello lws" (9 bytes)
[websocket-frame] encoded 15 bytes (2 header + 4 mask key + 9 payload)
[websocket-frame] decoded payload="hello lws" (9 bytes)
[websocket-frame] round trip OK
[websocket-frame] done
```

## Hardware swap

This example has no board-specific wiring -- it's pure buffer logic.
Once wired to a real libwebsockets `lws_context` over a TCP/TLS
socket, the transport is what changes; the frame layout this example
teaches stays the same regardless of SoM family.
