## 1. Hardware values marked `TBD`

These are the `metadata/e1m_modules/<SKU>.yaml` entries that intentionally
carry placeholder strings.  The orchestrator's carve-out resolver records
these as `status: blocked` + `reason: "memory_map.base is TBD for <region>"`
in the manifest and emits an `#error` directive in
`<alp/system_ipc.h>` — so `west alp-build --emit system-manifest`
succeeds (CI manifest-shape + determinism gates stay green) but the
actual slice-build trips at compile time until the values land.  Until
they're filled, the AEN + iMX93 examples build the C side fine but the
slice build refuses to link.

## 2. Implementation decisions made on your behalf

The spec covered the architecture but left judgement calls to the
implementer.  Here's every non-trivial one I locked in.

### 2.1 Core ID nomenclature

**Where:** spec §3.3 + `metadata/socs/.../*.json cores[].id` + `topology:`
keys across `metadata/e1m_modules/E1M-*.yaml`.

**Decision:**

- V2N's on-die M33 with `subtype: "system-manager"` → `m33_sm`.
- Standard iMX93 M33 (no subtype) → `m33`.
- Alif M55-HP / M55-HE → `m55_hp` / `m55_he` (matches Alif's own naming).
- Cortex-A clusters → `<arch>_cluster` (e.g. `a55_cluster`, `a32_cluster`)
  because a cluster shares an MMU + Linux instance — one config slot
  covers them all.

**What you'd change:** if customer-facing docs prefer different IDs
(e.g. `cortex_m33_sm` instead of `m33_sm`), one global rename
across the repo + a regression test pass.

**Why this:** terse but unambiguous; matches Zephyr's existing core
identifier conventions.

### 2.2 Default OS per core class

**Where:** spec §3.2 + every SoM preset's `topology:` block.

**Decision:**

- Cortex-A (A55, A32, A53) → `os: yocto`, default app `alp-image-edge`.
- Cortex-M (M33, M55, M7) → `os: zephyr`, default app `alp-stock-shim`
  (the SDK-owned idle image under `firmware/alp-stock-shim/`, unless
  `cores.<id>.app` is overridden).
- A bare `som: { sku: E1M-V2N101 }` produces both A55=Yocto and
  M33-SM=Zephyr.  Opt-out is explicit (`os: off`).

**What you'd change:** if "Zephyr-only on M-class is the default" framing
ever needs to flip (e.g. bare-metal becomes default on E3/E4 because
they're memory-constrained), rewrite the topology blocks accordingly.

**Why this:** matches the user's explicit "Zephyr + Yocto enabled by
default" directive given mid-design.

### 2.3 RPMsg endpoint ID derivation

**Where:** `scripts/alp_orchestrate/` + spec §6.3.

**Decision:**

- `src_ept = 0x400 | (FNV-1a(ipc.name) & 0x0FF)`.
- `dst_ept = src_ept + 1`.
- Collision-checked at emit time (loader error if two `ipc[]` entries
  hash to the same low byte).

**What you'd change:** if collisions ever happen in customer projects,
extend to 16-bit suffix or move to an explicit `endpoint_ids:` block in
`board.yaml`.

**Why this:** deterministic + readable IDs; spec called for it.

### 2.4 Carve-out region selection algorithm

**Where:** `scripts/alp_orchestrate/`.

**Decision:**

1. Sort `ipc[]` entries alphabetically by `name`.
2. Filter `memory_map[]` regions whose `accessible_from:` covers all
   endpoints.
3. Prefer non-cacheable regions (V2N's M33-SM has no cache).
4. Among acceptable regions, **prefer smallest first** (so OCRAM beats
   DDR for default 512 KiB carve-outs).
5. Allocate top-down within the chosen region.

**What you'd change:** if "prefer smallest" causes capacity issues
(small region fills up before all IPC entries land), flip to a best-fit
algorithm OR let `board.yaml ipc[]` carry an explicit `region:` hint.

**Why this:** keeps DDR free for application memory; OCRAM is fast +
non-cacheable so it's the natural IPC home.

### 2.5 CC3501E flash backend tool

**Where:** `scripts/flash_backends/cc3501e_usb_bootloader.py` (Wave 5B
landed).

**Decision (final):** backend is structurally complete — `name`,
`requires`, `flash_args` contract (documented in the module docstring of
`scripts/flash_backends/cc3501e_usb_bootloader.py`), and dry-run path
all final.  The real-invocation path raises
`NotImplementedError` with a TODO reference.  The dispatcher in
`alp_flash.py` catches that exception and logs/skips rather than
failing the entire flash flow.

**What you'll change when TI publishes the tool:** swap the
`NotImplementedError` body for a real `subprocess.run([...])`
invocation.  The `flash_args` already covers `device`, `mode`
(`otp_program` / `ram_load`), `speed`, `verify`.

**Why this:** SWRU626 documents the wire protocol but TI hasn't
published the host CLI wrapper.  Faking a tool name would lie to
customers.

### 2.6 Linux `alp_rpc_call` serialization model

**Where:** `src/yocto/rpc_yocto.c` (Wave 5A — landed).

**Decision (final):** **per-channel serialization**, matching the
Zephyr backend exactly.  Peer echoes the same method name in the
response; RX worker matches by `strncmp(method, ch->call_method, ...)`
while `ch->call_pending` is true.  Concurrent calls on one channel
block on `tx_mutex` + a separate `call_mutex` + `pthread_cond_t` so the
caller can wait without holding `tx_mutex` (avoids deadlocking the RX
worker).

**What you'd change:** if v0.7 needs concurrent calls, extend the
framing protocol (prepend a 32-bit correlation_id to the method header)
+ update both Zephyr and Linux backends in lockstep.  The single-call
serialization is documented in the header `@note`.

**Known divergence (flag for v0.7):** Zephyr's `alp_rpc_call` drops
the partial copy on `ALP_ERR_NOMEM`; Linux's copies what fits.  The
header doc doesn't pin one behaviour.  Align in v0.7.

**Why this:** simplest backend that matches the Zephyr-side contract.
Concurrent calls were not in the v0.6 acceptance criteria.

### 2.7 Helper-MCU naming on V2N

**Where:** `metadata/e1m_modules/E1M-V2N*.yaml helper_firmware[].chip`.

**Decision:** chip slug is `gd32_bridge`, not `gd32g553`.  Wave 3 agent
chose this to match the firmware build output directory
(`firmware/gd32-bridge/`).  The actual silicon part number stays
`gd32g553` in the `on_module.supervisor_mcu:` field — only the slug
in `helper_firmware:` is rename-flavoured.

**What you'd change:** if customer-facing docs reference the silicon
part number consistently, align by renaming the slug to `gd32g553` OR
adding both keys.

**Why this:** the `helper_firmware` entries name *artefacts*, not
silicon; the bridge firmware is one of several things the GD32 hosts.

### 2.8 Yocto MACHINE naming convention

**Where:** `meta-alp-sdk/conf/machine/*.conf` (Wave 5C in flight is
renaming) + `metadata/e1m_modules/*.yaml topology.<core>.machine`.

**Decision:** per-cluster naming: `e1m-<sku>-a55` for the A-side of a
V2N or iMX93 SoM.  AEN A32-cluster MACHINEs are deferred to v0.7
(per Phase 5 CI's `pr-bitbake.yml` comment) because AEN's Linux story
isn't a priority yet.

**What you'd change:** when AEN's A32-cluster Linux flow becomes a
target, add `e1m-aen<sku>-a32.conf` + corresponding `pr-bitbake.yml`
matrix entries.

**Why this:** A-cluster MACHINEs are the only ones Yocto sees (M-class
slices ride west, not bitbake).

### 2.9 ADR 0010's "alternatives considered" list

**Where:** `docs/adr/0010-heterogeneous-os-orchestration.md`.

**Decision:** three alternatives recorded as rejected: V2N-only-first,
backwards-compatible dual schema, mproc-primitives-only.

**What you'd change:** if you remember rejecting additional approaches
during the actual decision-making (Kas-style superbuild, dedicated
`alp` CLI separate from west, etc. — those came up in the brainstorm),
add rows to the ADR's rejected-alternatives table for completeness.

**Why this:** the brainstorm captured these alternatives but the ADR
agent included the three most-load-bearing ones to keep ADR concise.

## 6.4 Wave 5B — flash backend safety gates + flash_args extensions

**Where:** `scripts/flash_backends/*.py`.

The flash backends added safety mechanisms + per-backend `flash_args`
fields beyond what the spec called for.  None breaks existing manifest
shape — they're additive — but they affect operator UX.

| Backend | Safety mechanism | Bypass |
|---|---|---|
| `yocto_wic_to_sd_or_emmc` (alias: `yocto_wic`) | `flash_args.confirm: true` required to actually write; otherwise treats the call as a dry-run regardless of `--dry-run` flag | `flash_args.confirm: true` in manifest OR `ALP_FLASH_FORCE=1` env var |

| Backend | Extra `flash_args` accepted | Default |
|---|---|---|
| `yocto_wic` | `confirm`, `bs` | `confirm: false`, `bs: "4M"` |
| `zephyr_west_flash` | `erase`, `hex_file` | `erase: false`, `hex_file: null` |
| `baremetal_cmake_flash` | `jobs`, `config` | `jobs: $(nproc)`, `config: null` |
| `swd_probe` | `reset`, `use_pyocd` | `reset: true`, `use_pyocd: false` |
| `cc3501e_usb_bootloader` | `speed`, `verify` | `speed: 921600`, `verify: true` (auto for `otp_program`) |

**What you might want to revisit:**

- The `confirm` gate on `yocto_wic` is restrictive but prevents
  accidental `/dev/sda` writes.  If your manufacturing flow needs
  unattended flashing, set `ALP_FLASH_FORCE=1` in the fixture's env.
- The `swd_probe.use_pyocd` toggle was added so CI can use pyocd
  (pip-installable) while bench engineers use openocd.  Pick one as
  the canonical CI default in `pr-bitbake.yml` when that wave lands.

## 6.5 Wave 5A — Linux backend quirks that may bite later

**Where:** `src/yocto/rpc_yocto.c` + `src/yocto/CMakeLists.txt`.

| Quirk | Impact | Fix when relevant |
|---|---|---|
| `pthread_cond_timedwait` uses `CLOCK_REALTIME` by default on glibc | Wall-clock jumps (NTP sync, manual `date -s`) can shift the deadline mid-wait | If SDK-wide convention adopts `CLOCK_MONOTONIC`, set `pthread_condattr_setclock(CLOCK_MONOTONIC)` at channel open. |
| `RPMSG_CREATE_EPT_IOCTL` from `<linux/rpmsg.h>` is the kernel ABI | Kernels < 4.18 lack this IOCTL | The CMake already requires `meta-openamp`-class libs but doesn't pin a kernel version; document the floor in a header comment + add a `pkg-config` kernel-headers check. |
| `clock_gettime` needs `-lrt` on glibc < 2.17 | Won't link on a pre-glibc-2.17 sysroot | Add `-lrt` to `src/yocto/CMakeLists.txt`'s `target_link_libraries` conditionally on glibc version. |
| `/dev/rpmsg0` chardev selection is hard-coded with `ALP_RPMSG_DEV` env override | Multi-channel hosts can collide | Walk `/sys/class/rpmsg/rpmsg*/name` for the matching endpoint name; punted from Phase 3 + Wave 5A. |
| Live tests (5 of them in `TestRpcYoctoLive`) are skipped behind `_have_openamp_user()` gate | No actual rpmsg-userspace coverage in CI | Either `LD_PRELOAD`-shim `open`/`ioctl`/`read`/`write` for the chardev paths, or run live tests on the self-hosted Yocto runner once HIL ships. |

## 7. Things that landed but might want a sanity re-review

- **`scripts/alp_orchestrate/`** — 1295 lines, ProcessPoolExecutor
  fan-out, Windows-tolerant skip-on-missing-tools.  Test coverage is
  14 cases; some Phase-3-introduced surface (helper_mcus, flash_method
  population) got 3 specific tests but the broader integration path
  could use a few more.
- **`<alp/rpc.h>`** — API surface is final but the framing format
  (method-name header + opaque payload) is deliberately minimal.  v0.7
  may want a richer schema (length-prefixed binary or protobuf).
- **DT overlay include path** — every `zephyr/boards/<sku>_<core>.overlay`
  uses `#include "../../build/generated/dts-reservations.dtsi"`.  This
  relative path assumes the overlay is included from
  `zephyr/boards/<...>/<overlay>` which it is, but if Zephyr ever
  rearranges board discovery, this breaks.  Phase 4 follow-up noted.
- **Five-phase commit story** — v0.6 ships across 6 commits on `main`
  (spec + 5 phases).  If you ever rebase + want this as a single squashed
  v0.6 commit, the history's already linear so it's a clean operation.
