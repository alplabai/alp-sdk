# 0018. Curated third-party libraries: one `libraries:` flag, capability-gated, never forked

Status: Accepted
Date: 2026-07-03

alp-sdk's next developer-experience step after backend completeness is
letting customers pull common third-party libraries (GUI, DSP, cloud,
robotics, scripting) into a project with **one declaration** — without
hand-wiring Kconfig, west manifests, Yocto recipes, or CMake per OS and
per family. This ADR defines the mechanism, the curation policy, and
the boundary against becoming a package manager or a fork farm.

## Context

- Applications declare their build through the project YAML, and every
  build artefact is **emitted** from it (ADR 0014): Kconfig fragments,
  cmake-args, Yocto config, board files. There is deliberately no
  second place where build configuration lives.
- The capability layer (`ALP_HAS_*` / `alp_has()`, generated by
  `scripts/gen_soc_caps.py` from `metadata/`) already answers "does
  this SoM/core have a display / an NPU / enough flash" — and the
  portability matrix is now generated from the same sources.
- ADR 0017 fixes the posture toward external code: alp-sdk integrates
  and pins upstream/vendor libraries; it does not rewrite or fork them.
- LVGL is already consumed as a re-export and proves the demand: the
  matrix advertises it per-column, but enabling it today is
  hand-assembled per project.
- alp-studio (ADR 0005) needs a machine-readable answer to "which
  libraries can this board run" to offer them in its UI. It must get
  that from SDK metadata, not from its own list.

## Decision

**One declaration in the project YAML selects libraries; everything
else is emitted and validated:**

```yaml
libraries: [lvgl, cmsis-dsp, nanopb]
```

1. **Library manifests live in `metadata/libraries/<name>.yaml`** —
   one file per library (single source of truth, like chips and SoM
   presets). A manifest declares:
   - `integration:` per-OS wiring — Zephyr (Kconfig symbols + west
     module pin), Yocto (`IMAGE_INSTALL` / recipe include), baremetal
     (CMake package/FetchContent pin), each optional;
   - `requires:` capability constraints resolved against the existing
     capability layer (e.g. `display`, `ethos_u`, `min_flash_kib`,
     `min_ram_kib`, `core_class: m|a`, `os: zephyr|yocto`);
   - `tier:` curation tier (below);
   - `version:` the pinned upstream version;
   - `license:` SPDX id (gate: see non-goals).
2. **The orchestrator emits the wiring** (`--emit` grows the library
   fragments) exactly as it does for boards today — no new build
   entry point, no shadow configuration path (ADR 0014 applies).
3. **Compatibility is validated at emit time** from `requires:` vs the
   resolved SoM/core capabilities. An incompatible selection fails
   with the same clear-error contract as schema validation —
   `alp doctor` and alp-studio read the same manifests, so the IDE
   filter and the CLI error can never disagree.
4. **Two curation tiers** bound the CI cost:
   - **Tier A — curated**: version-pinned, built in CI for at least
     one board per supported family, ships one teaching example.
     Breakage blocks release.
   - **Tier B — recipe-only**: wiring + compatibility metadata are
     maintained and emitted, but the library is not built in alp-sdk
     CI; the manifest says so, and `alp doctor` labels it.
5. **ADR 0017 extends to libraries verbatim**: alp-sdk pins and
   integrates upstream releases. No forks, no vendored copies of
   library sources in-tree, no alp-sdk-only patches beyond build-glue
   that upstream would reject. A library that cannot be consumed that
   way is not integrated.

### Initial curation (priority order)

Prove the mechanism on already-consumed, low-effort entries, then add
the differentiators:

1. **LVGL** (Tier A) — formalises the existing re-export as the first
   manifest; zero new upstream surface.
2. **CMSIS-DSP / CMSIS-NN** (Tier A) — M55 Helium story; already in
   the Zephyr module set.
3. **nanopb + zcbor** (Tier A) — serialization for every cloud/IPC
   customer; tiny CI cost.
4. **micro-ROS (M-class) + ROS 2 (A55/Yocto)** (Tier A, flagship) —
   the heterogeneous-core robotics story on V2N/AEN that ADR 0010's
   Zephyr+Yocto peering enables; exercises the mechanism across both
   OSes in one project.
5. **Cloud/device services** (Tier B first: AWS IoT embedded-C, Azure
   IoT, Memfault, LwM2M/CoAP via Zephyr) — promote per demand.
6. **MicroPython** (Tier B) — high eval value, heavy maintenance;
   promotion to Tier A requires a dedicated owner.
7. **Industrial** (CANopenNode Tier B, Zephyr Modbus Tier A-cheap) —
   CAN is already GA.

## Alternatives considered

- **Do nothing (document per-library recipes).** Keeps the SDK small,
  but every customer re-derives the same Kconfig/west/Yocto wiring,
  and studio has no machine-readable compatibility source. Rejected:
  the whole SDK premise is emitting exactly this kind of glue.
- **A real package manager (west-only, or a custom `alp pkg`).**
  Over-scoped: west already pins Zephyr-side modules and Yocto already
  has layers; the missing piece is unified *selection + compatibility
  + emission*, not fetching. Rejected.
- **Per-library Kconfig only (no metadata).** Works for Zephyr,
  invisible to Yocto/baremetal and to studio, and splits the truth
  between Kconfig and YAML. Rejected on the single-source principle.
- **Vendoring library sources in-tree.** Explicitly forbidden by
  ADR 0017's rationale (maintenance liability, licence tangle,
  divergence risk). Rejected.

## Consequences

- New schema + directory (`metadata/libraries/`), a validation pass in
  `validate_metadata.py`, emitter growth in the orchestrator, and an
  `alp doctor` section. Each is an extension of an existing pattern,
  not a new subsystem.
- CI grows one Tier-A build lane per family; Tier B costs nothing in
  CI. The tier split is the knob if CI time becomes a problem.
- The portability matrix generator gains a libraries table for free
  (same metadata), so the docs answer "which libraries run where"
  without hand maintenance.
- alp-studio's library picker becomes a thin view over
  `metadata/libraries/` (ADR 0005 boundary preserved).
- **Non-goals:** redistributing licence-gated SDKs or binaries
  (manifests may *reference* a vendor download the customer performs —
  same rule as BSPs); GPL-incompatible licence surprises (the
  `license:` field is validated against an allowlist and surfaces in
  `alp doctor`); guaranteeing every library on every core (the
  `requires:` block exists precisely to say no early and clearly).
