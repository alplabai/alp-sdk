## Pillar 1 — Code completion

### 1z. E1M instance-ID rename + pin-as-GPIO fallback (CRITICAL, pre-ABI-freeze)

**Captured 2026-05-14 from maintainer:** the instance identifiers
formerly in `<alp/e1m_pinout.h>` (then `ALP_E1M_*`) carried
redundant prefix + over-granular channel selection.  Three
changes required before v1.0 ABI freezes the surface:

1. ✅ **Drop `ALP_` prefix (DONE, this session).**  Old form
   `ALP_E1M_I2C0` → new form `E1M_I2C0`; `ALP_E1M_PWM3` →
   `E1M_PWM3`; etc.  Rationale: **E1M is the open-standard
   form factor, not an ALP-specific concept** -- the prefix
   was redundant noise.  The `ALP_*` namespace stays
   reserved for SDK-internal abstractions (`ALP_OK`,
   `alp_status_t`, etc.).
2. ✅ **Collapse channel selection (DONE, this session).**  Old
   form `ALP_E1M_DAC_CH0` had a peer `ALP_E1M_DAC0` of identical
   value -- the `_CH<N>` aliases got removed so the pin-granular
   `E1M_DAC<N>` is the single name.  Rationale: **callers pick
   the pin and its matching peripheral mode**; the
   pin-to-channel routing is part of the E1M-spec mapping, not
   a per-call choice.  Confirmed no `_CH<N>` suffixes survived
   on any other peripheral (`E1M_PWM*` / `E1M_ADC*` /
   `E1M_ENC*` were already pin-granular).
3. ✅ **Pin-as-GPIO secondary (shipped).**  Any pin that supports
   ADC / PWM / QENC is also openable as a plain GPIO via its
   GPIO-secondary index `E1M_GPIO_<class><N>`.  The two ID spaces
   stay deliberately separate: the peripheral-instance space
   (`E1M_ADC0..7` = 0..7) selects the analog/timer function, while
   the GPIO-index space (`E1M_GPIO_IO0..25` = 0..25,
   `E1M_GPIO_PWM0..7` = 26..33, `E1M_GPIO_ADC0..7` = 42..49, ...)
   drives the same pad digitally -- e.g. `alp_gpio_open(E1M_GPIO_ADC0)`
   opens ADC0's pad as GPIO.  No silent unification; the caller names
   the role explicitly (decision resolved -- was the v0.x "fallback" TBD).

### 1b. V2N supervisor → DA9292 DEEPX rail

> **Status:** design call landed (supervisor-owns-shared-handle);
> implementation in §C.28.

- [x] **BRD_I²C multi-consumer pattern** — the V2N supervisor singleton owns the `alp_i2c_t` and exposes `alp_z_v2n_supervisor_brd_i2c_acquire()` / `_release()` borrowing pair that shares the same mutex as the existing GD32 acquire helper. - [x] **`src/zephyr/v2n_power_mgmt.c`** — P65 rising-edge IRQ → workqueue → borrow BRD_I²C → `da9292_v2n_m1_enable_deepx_rail` (CH2=0.75 V with the VSTEP=1 trap handled, polls CH2 PG up to 5 ms) → release I²C → drive P64 high.  `SYS_INIT` at APPLICATION priority runs `da9292_v2n_base_init` so the DA9292 ctx is hot before the first edge.  Compiles in only when `CONFIG_ALP_SDK_V2N_POWER_MGMT=y` + `CONFIG_ALP_SDK_V2N_SUPERVISOR=y`; defaults on for V2N + DA9292 builds.  Landed §C.28.
