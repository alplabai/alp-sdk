## Pillar 1 — Code completion

### 1z. E1M instance-ID rename + pin-as-GPIO fallback (CRITICAL, pre-ABI-freeze)

**Captured 2026-05-14 from maintainer:** the instance identifiers
formerly in `<alp/e1m_pinout.h>` (then `ALP_E1M_*`) carried
redundant prefix + over-granular channel selection.  Three
changes required before v1.0 ABI freezes the surface:

1. ✅ **Drop `ALP_` prefix (DONE, this session).**  Old form
   `ALP_E1M_I2C0` → new form `E1M_I2C0`; `ALP_E1M_PWM3` →
   `E1M_PWM3`; etc.  Rationale: **E1M is the open-standard
   form factor, not an ALP-specific concept** -- the prefix
   was redundant noise.  The `ALP_*` namespace stays
   reserved for SDK-internal abstractions (`ALP_OK`,
   `alp_status_t`, etc.).
2. ✅ **Collapse channel selection (DONE, this session).**  Old
   form `ALP_E1M_DAC_CH0` had a peer `ALP_E1M_DAC0` of identical
   value -- the `_CH<N>` aliases got removed so the pin-granular
   `E1M_DAC<N>` is the single name.  Rationale: **callers pick
   the pin and its matching peripheral mode**; the
   pin-to-channel routing is part of the E1M-spec mapping, not
   a per-call choice.  Confirmed no `_CH<N>` suffixes survived
   on any other peripheral (`E1M_PWM*` / `E1M_ADC*` /
   `E1M_ENC*` were already pin-granular).
3. 📋 **Pin-as-GPIO fallback (TBD).**  Any pin that supports
   ADC / PWM / QENC must also be openable as GPIO.  If a
   customer's app doesn't need the analog/timer function of a
   pin, they should be able to `alp_gpio_open(E1M_ADC0)` and
   get a usable digital GPIO with no error.  Today the
   peripheral-instance ID space (`E1M_ADC0..7` = 0..7) and
   the GPIO-index ID space (`E1M_GPIO_IO0..25` = 0..25,
   `E1M_GPIO_PWM0..7` = 26..33, ...) are separate; the
   fallback either unifies them or adds an explicit
   conversion in `alp_gpio_open`.  Decision TBD.

**Scope (item 1 only -- items 2 + 3 deferred to next commit):**

- `include/alp/e1m_pinout.h` -- header guard + every macro
  renamed.
- Every example's `main.c` that uses an instance ID
  (~25 examples).
- Every test file under `tests/zephyr/peripheral/` and
  `tests/zephyr/chips/`.
- Every `src/zephyr/peripheral_*.c` backend that consumes the
  IDs.
- `metadata/templates/board.yaml` if it cites IDs.
- `docs/board-config.md`, `docs/glossary.md`,
  `docs/getting-started.md`, every tutorial / example README.
- `docs/abi-markers.md` -- regenerate `<alp/e1m_pinout.h>`'s
  ABI status entry once the rename lands.

### 1a. GD32 firmware HAL bodies (wave-2 opcodes)

Every wave-2 reserved opcode in `firmware/gd32-bridge/hal/` returns
`STATUS_NOSUPPORT` today.  The host-side `<alp/*>` surface is wired
in v0.5; firmware bodies are the missing half.

> **Status:** writeable now (HAL library reachable at
> [`alplabai/gd32g5x3-firmware-library`](https://github.com/alplabai/gd32g5x3-firmware-library));
> verification on real GD32G553 silicon still HiL-gated.  Bodies
> can land + CI-build under the existing `pr-gd32-bridge-build.yml`
> matrix; flip from 🟡 to ✅ once the HiL rig exercises them.

- 🟡 **`bridge_hw_pwm_capture_begin/read/end`** (opcodes `0x23` / `0x24` / `0x25`) — GD32 timer input-capture mode on the eight PWM channels.  Bodies landed §C.15a (polled drain + state machine + ns conversion); HiL flip pending pad routing rework (V2N PWMs use the CHxN pad, capture path uses CHx).
- 🔌 **`bridge_hw_pwm_single_pulse`** (opcode `0x26`) — already wired to OPM mode; verify completeness against `<alp/pwm.h>`.
- 🟡 **`bridge_hw_timer_sync`** (opcode `0x27`) — master/slave timer sync via TIMx_SMCR.  Body landed §C.15b (master/slave id 0..2 → TIMER0/7/19; mode 0..5 → SMC encoding via switch; master TRGO0=UPDATE; slave listens to ITI0).
- 🟡 **`bridge_hw_power_mode_set`** (opcode `0x28`) — Sleep / Deep-Sleep / Standby via PMU registers.  Modes 2 + 3 landed §C.15c (`pmu_to_deepsleepmode` / `pmu_to_standbymode`); wake-source bitmap partial-applied (GPIO bit enables WAKEUP_PIN0..4; other bits + `wake_after_ms` still NOSUPPORT pending RTC alarm).
- 🟡 **`bridge_hw_adc_stream_configure_dsp`** (opcode `0x36`) — chain ADC stream output through the on-firmware DSP slot.  `chain_open` + `stage_push` landed §C.15d (4-chain pool + per-stage chunk reassembly into a 260-byte buffer + ordering / kind / completion guards); `chain_bind` still NOSUPPORT pending the ADC_STREAM_BEGIN / DMA stream-attach plumbing.
